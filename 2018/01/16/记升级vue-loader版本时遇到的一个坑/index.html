记升级vue-loader版本时遇到的一个坑

2018-01-16

最近对公司的一个Vue项目进行了依赖升级, 主要是将webpack1升级到了webpack3, 同时也将原本项目的Vue、Vue-router、Vuex、Vue-loader等一系列依赖进行了升级, 升级过程中的相关配置参考了Vue-cli中webpack的template, 主要是在其上做了一些适应我们项目的调整, 整体过程还是很轻松愉快的, 但是在整个过程中也遇到了一些小坑, 主要集中在单元测试这一块, 比如webpack2开始不支持loader的省略写法呀、覆盖率用babel配合istanbul而不需要像webpack1里过多的手动配置呀, 网上的相关webpack升级指南教程之类的文章博客很多, 这里就不具体涉及了, 只是记录与vue-loader版本升级有关的一个隐蔽的坑。

当项目整体升级完毕后, 业务代码跑起来正常, 然后再跑单元测试, 结果报了许多原本没有出现的断言错误, 深入其中发现其每次断言错误都会伴随着一句Vue警告

初步猜测可能是我们测试中没有正确require到我们被测试的相关组件, 在测试中我们一般是通过类似以下形式去拿到我们被测试的组件并进行依赖注入的

此时在出错的测试块中, 通过console打印我们依赖注入的组件comp, 打印的结果正如Vue警告所示, 并没有template或render方法

为什么require进来的组件没有template或者render呢, 正常来说是应该具备的, Vue单文件组件中<template></template>里的模板, 在经过vue-loader加载后理应会被正确导入为render函数, 这里却出现了遗漏。
初步判断问题与vue-loader有关, 进入github上vue-loader的repo, 在其issue页面, 查找我们的遇到的vue警告内容, 发现类似的问题issue996。

噢, 原来是vue-loader@13.0.0版本中的Breaking Changes引起的。
如vue-loader@13.0.0的release所描述

The esModule option is now true by default, because this is necessary for ES-module-based scope hoisting to work. This means the export from a *.vue file is now an ES module by default, so async components via dynamic import like this will break:

const Foo = () => import('./Foo.vue')
Note: the above can continue to work with Vue 2.4 + vue-router 2.7, which will automatically resolve ES modules’ default exports when dealing with async components. In earlier versions of Vue and vue-router you will have to do this:

const Foo = () => import('./Foo.vue').then(m => m.default)
Alternatively, you can turn off the new behavior by explicitly using esModule: false in vue-loader options.

Similarly, old CommonJS-style requires will also need to be updated:

// before
const Foo = require('./Foo.vue')

// after
const Foo = require('./Foo.vue').default

原来, 在vue-loader@13.0.0中, 为了支持webpack3中的基于esModule的Scope Hoisting(知乎上有篇相关文章写的很详细), esModule选项被默认设置为true。

知道了问题产生的原因, 那解决办法也就相应出来了。大致是俩种。

  • 第一种 更改我们测试里的require语法, 将const inject = require('!!vue-loader?inject!src/componentA.vue')修改为const inject = require('!!vue-loader?inject!src/componentA.vue').default

  • 第二种 修改测试的config, 在测试环境时, 将vue-loader的esModule选项设置为false

因为我们的测试文件比较多, 采用第一种方法, 就算用编辑器(如sublime)的全局replace, 也要手动去一个一个保存, 较为繁琐, 而手动写一个node脚本处理也有一定的时间成本。所以这里我选择采用第二种解决方法, 直接修改测试环境的配置, 很简单, 只需要在我们的webpack.test.conf.js中加上一段代码处理就OK：
