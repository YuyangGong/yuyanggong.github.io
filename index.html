karma单元测试中用ChromeHeadless取代PhantomJS

2018-02-01

在项目中, 进行自动化测试的时候, 一般使用的无头浏览器是PhantomJS, 但是其速度较慢, 而且有着一些稀奇古怪的小坑, 有部分可以通过小hack绕过或模拟出效果, 还有部分就比较麻烦,比如近期我遇到的一个小bug, 进行了事件委托的元素, 无论是调用类似HTMLElement.click的原生方法, 还是通过document.createEvent创建自定义事件然后通过dispatchEvent触发, 在PhantomJs中都无法正确触发事件, 得不到我们想要的效果, 这种情况下, 可能我们在写测试代码就需要通过stub去模拟拿到我们的事件监听的回调, 然后手动执行回调进行测试了, 很不优雅且测试的逻辑也不完整。想起前几个月Chrome已经发布了Headless版本, 那么我们为什么不用ChromeHeadless来取代PhantomJs进行测试呢?
完全是可以的, 其过程也很简单, 分为以下4个步骤

  1. 修改package.json中的依赖
  2. 修改karma的配置文件
  3. 去除测试suit中为PhantomJs所写的hack code
  4. 修改持续集成配置文件(在gitlab中是gitlab-ci.yml)

字符编码拾遗

2018-01-24

最近遇到一个需求, 对用户的输入长度进行限制, 这就需要判断单双字节，而我对这一块比较模糊，便做了点总结。

单字节：

一种字符编码方式，每个字符用一个字节(byte)表示。所以单字节字符集不可能包含256(2^8)个以上 的字符。单字节字符包含拉丁文字母表，重音字符（accented characters）及ASCII标准和DOS操作系统定义的图形字符。ASCII码就为单字节字符集的一种。

双字节：

双字节字符集（DBCS：double-byte character set）,解决中国、日本和韩国的象形文字符和ASCII的某种兼容性。
DBCS从256代码开始，就像ASCII一样。与任何行为良好的代码页一样，最初的128个代码是ASCII。
然而，较高的128个代码中的某些总是跟随着第二个字节。
这两个字节一起（称作首字节和跟随字节）定义一个字符，通常是一个复杂的象形文字。
